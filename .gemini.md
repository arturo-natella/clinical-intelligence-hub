# MedPrep Development Standards

This document serves as the absolute foundational mandate for all AI-assisted development on the MedPrep application. These rules take precedence over general workflows to ensure privacy, local-first processing, and methodological consistency with the architectural vision.

## 1. Local-First / Privacy Mandate
- **Default to Local:** Always default to local processing using Ollama (MedGemma) and MONAI. Raw records (PDFs, DICOMs) must NEVER be sent to cloud APIs by default.
- **API Fallbacks:** Cloud APIs (Gemini 3.1 Pro via `google-genai`) are strictly fallbacks for documents the local models fail to parse. These are secondary options, not primary entry points.
- **Local Standardization:** Terminology standardization must use local databases (LOINC, SNOMED CT) before reaching out to external networks.

## 2. CK Plan Adherence
- **Architectural Single Source of Truth:** The file `/Users/owner/docs/plans/CK Plan.md` contains the authoritative architecture.
- **Mandatory Review:** Before creating new data processors or integrating APIs, you MUST review the tables in `CK Plan.md` to identify if the task is meant for a local model, a free clinical validation API (OpenFDA, RxNorm), a continuous monitoring daemon, or a Playwright scraper.
- **No Unplanned APIs:** Do not add new cloud APIs unless explicitly instructed by the user.

## 3. UI Preservation
- **Keep Additions:** The premium GUI additions created during prototyping (Drag & Drop Upload, Correlative Timeline, Visual Body Map, RAG Chat) are approved and must be preserved during backend refactoring.
- **Offline Capable:** The UI must fail gracefully if cloud APIs are not configured, relying on local inference and local databases as designed.
